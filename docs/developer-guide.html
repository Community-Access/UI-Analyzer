Introduction

UI Analyzer is a cross-platform wxPython app built for Becky, a blind developer who needs to understand what her UI looks like to sighted users. It uses a local AI model served by Ollama to describe UI source files — no cloud, no internet for analysis, no data leaves the machine.

The core loop: open a project folder → select a UI file or drop an image → get a structured written description of its visual appearance, usability, professional quality, and accessibility. Ask follow-up questions in a persistent conversation per file. Build a project-wide context and ask cross-screen questions.

This guide covers everything a developer new to the project needs to contribute without breaking existing behavior.

Feature overview

  • File analysis — Analyze Swift, SwiftUI, Storyboard, XIB, HTML, CSS, React/JSX, Vue, Svelte, Python, JavaScript, TypeScript, and Markdown files. Six-section output covers visual appearance, usability, professional quality, accessibility, other details, and screen layout.
  • Design spec analysis — Non-readme Markdown files are detected as design specs and routed to a color/typography/component-focused prompt covering the full token vocabulary.
  • Image / screenshot analysis — PNG, JPG, HEIC, WebP, TIFF, BMP, GIF up to 10 MB. Pillow measures dimensions; Tesseract OCR extracts text with positional labels; Ollama describes the screen.
  • Screenshot attachment — Any source file can have a .context.png screenshot attached. The screenshot is stored in the project folder and the binding is persisted in the platform user-data dir via platformdirs. Once attached, every analysis includes the screenshot via the vision pipeline.
  • Vision pipeline with automatic OCR fallback — Multimodal models receive image bytes directly (VisionMode.MULTIMODAL). If the model rejects images with a 400 error, Tesseract OCR runs on the screenshot and the transcript is injected as text (VisionMode.OCR_TEXT).
  • Validate against screenshot — Sends the prior analysis + attached screenshot to the model and asks it to stand by, retract, or extend each claim. Returns a structured ValidationResult with three lists. Presented in an accessible dialog with Copy support.
  • Framework-specific API references — Content-first detection picks one of twelve framework API reference strings (SwiftUI, React, React Native, Vue, HTML, wxPython, PyQt6, tkinter, WinForms, WPF, .NET MAUI, Avalonia) and injects it into the prompt so suggestions are actionable, not generic.
  • Automated contrast advisor — Extracts all hex colors from source, computes WCAG ratios, flags failures, and suggests fixes — all before the model sees the file, so the model can cite precise numbers.
  • Output modes — Prose (six-section Markdown with navigable headings), Markdown table, or a full accessible HTML5 document with TL;DR summary, element table, and layout table.
  • Follow-up questions — Persistent per-file conversation with automatic context rolling at 12,000 chars.
  • Project context — Generates 4-line summaries of all files so cross-screen questions work without re-analyzing. Project Q&A answers are formatted as HTML fragments with heading structure for NVDA/JAWS navigation.
  • Drag and drop — Drop folders from File Explorer or Finder onto the sidebar to load a project.
  • AI provider abstraction — AIClient protocol with three concrete implementations: OllamaClient (local), AnthropicClient (cloud), OpenAIClient (cloud). Switching providers requires only a config change, no code changes.
  • WCAG 2.2 AA accessible — Every control has an accessible name (SetName), every button is 44×44 px minimum, every dialog is keyboard-operable, and every screen reader (NVDA, JAWS, Narrator, VoiceOver) can read every screen.
  • Cross-platform — Windows 10/11 (Edge WebView2) and macOS 14+ (WebKit). Same codebase, same keyboard shortcuts (event.CmdDown() for Ctrl/Cmd), same feature set.

Ollama integration

UI Analyzer talks to Ollama over its local HTTP API at http://localhost:11434. The integration is in ui_analyzer/services/ollama_client.py.

Endpoints used

  • GET /api/tags — list installed models. Returns name, size, details.parameter_size, details.family.
  • POST /api/chat — streaming chat. Body has model, messages array, stream: true. The response is NDJSON; each line has message.content (a chunk of text), done: false while streaming, then done: true with stats on the final line.

Streaming pattern

async with client.stream("POST", "/api/chat", json=payload) as resp:
    async for line in resp.aiter_lines():
        chunk = json.loads(line)
        if chunk.get("done"):
            break
        on_chunk(chunk["message"]["content"])

All UI updates from inside the streaming loop must be wrapped in wx.CallAfter because the stream runs on a background thread. OllamaClient exposes a synchronous wrapper (chat_stream(...)) that runs the async loop in a worker thread and calls on_chunk from there.

Model recommendation

best_model_for_filetype(models, file_type) returns the best model for a given file type using a simple heuristic map (e.g. qwen2.5-coder:* for code, llava:* for images). The result is shown as (recommended) in the picker — the user can still pick anything they want.

Architecture and key design decisions

Three-layer structure

The app splits cleanly into models, services, and views:

  • models — plain dataclasses / enums. UIFile holds a scanned file's metadata and analysis state. UIAnalysis holds the model's response (text or HTML). UIFileType enumerates every supported file type with an is_image flag and an extensions tuple.
  • services — pure business logic with no wx imports. UIAnalyzer builds prompts, manages per-file sessions, and exposes a streaming analyze(...) method. OllamaClient handles all HTTP. file_scanner walks a folder. contrast_advisor computes WCAG ratios.
  • views — the wx widgets only. MainFrame holds the splitter, menus, and status bar. DetailPanel and SidebarPanel are children of the splitter. ModelPickerDialog is a modal dialog.

This split lets the services be unit-tested without wx, and lets the views be redesigned without touching the analysis logic.

Per-file conversation sessions

When a follow-up question is asked on a file, UIAnalyzer looks up (or creates) a FileSession keyed by the file's UUID. The session holds the rolling message history: system prompt, prior user/assistant turns, and the new question. Streaming tokens are appended to the running text in DetailPanel.stream_chunk.

Rolling context window

When the running message history exceeds 12 000 characters, roll_context(file_id) is called before the next follow-up. It asks the model for a 4–6 sentence summary, creates a fresh session with the system prompt plus the summary as the new system block, and re-anchors with the first 2 000 characters of the original file source.

Project context

build_project_context(files, on_progress) runs a lightweight mini-session for each file, generating a 2–3 sentence summary. All summaries are concatenated and stored in self.project_context. Every call to _make_session() appends this block to the system instructions, giving the model cross-file awareness. Project Q&A uses a separate _make_project_system_prompt that emits HTML fragment output with <h2>Question</h2> / <h2>Answer</h2> structure for screen-reader heading navigation.

HTML table rendering

When the user picks Table > HTML, the model is asked to return a self-contained HTML fragment starting with a <details class="tldr" open> TL;DR block, followed by a <table> with <thead>, <tbody>, and <th scope>. _extract_html(raw) strips any Markdown code fences the model may have wrapped around it, then DetailPanel._display_analysis hands the result to AccessibleWebView.set_content(html) (or the plain wx.html2.WebView.SetPage fallback).

The TL;DR <details open> element is the first thing in the output. Screen readers announce it as a collapsible region with the TL;DR text inside, so a blind user gets the headline before diving into the table.

Threading model

wxPython is single-threaded — all UI work must happen on the main thread. Long-running operations (folder scan, Ollama HTTP request, image OCR) run on background threading.Thread workers and post results back via wx.CallAfter.

The pattern is the same everywhere:

def _start_analyze(self, file, mode, fmt):
    self._status.SetStatusText(f"Analyzing {file.name}…")
    def do_analysis():
        try:
            analysis = self._analyzer.analyze(file, mode, fmt, on_chunk)
            wx.CallAfter(self._on_analysis_done, file, analysis)
        except Exception as exc:
            wx.CallAfter(self._on_analysis_error, file, str(exc))
    threading.Thread(target=do_analysis, daemon=True).start()

The on_chunk callback inside do_analysis is called from the worker thread, so UIAnalyzer.analyze wraps it: every chunk path inside the analyzer calls wx.CallAfter(self._detail.stream_chunk, text) before touching the wx widget. This is the only safe way to update a wx control from a background thread.

Never call self._status.SetStatusText(...), self._list.SetItem(...), or any other wx method directly from inside do_analysis — it works on a lucky run, then crashes randomly. Always go through wx.CallAfter.

Image analysis

Supported formats: PNG, JPG, JPEG, HEIC, HEIF, WebP, TIFF, TIF, BMP, GIF. Maximum size: 10 MB (enforced in file_scanner).

When file.file_type.is_image is true, UIAnalyzer.analyze calls _extract_image_content(path) before building the prompt:

  1. Dimensions and device type — Pillow opens the image and reports width × height. The aspect ratio is compared to common device screen ratios to produce a heuristic device label (e.g., "iPhone portrait 393×852 pt").
  2. OCR — pytesseract.image_to_data processes the image. Each word's bounding box is mapped to a position label: one of top/upper/middle/lower/bottom × left/center/right. Capped at 60 observations to keep the prompt small.
  3. Prompt — The header and OCR text are passed to _image_prompt, which asks the model to describe the screen visually using that structured text.

Ollama does not receive image pixel data directly — only structured text. Text-heavy screens work very well. Icon-heavy screens with no labels are harder; the model works from position information alone.

If Tesseract is not installed, pytesseract raises a clear error at runtime. The scanner still picks up image files; the user just gets an error when they try to analyze one. The error pane in DetailPanel shows the missing-binary message.

Vision analysis pipeline

When a screenshot is attached to a source file (see Screenshot attachment), the analysis pipeline tries three strategies in order, automatically falling back if the model does not support the previous strategy:

  1. Multimodal (MULTIMODAL) — The screenshot is base64-encoded and sent in the images field of the Ollama chat message alongside the source code. Vision-capable models (e.g., llama3.2-vision) receive the raw image pixels and can see the actual rendered UI. VisionMode.MULTIMODAL is recorded on the returned UIAnalysis.
  2. OCR text fallback (OCR_TEXT) — If the model returns an HTTP 400 or mentions "images" in the error, the pipeline removes the image from the message, runs Tesseract OCR on the screenshot, and injects the resulting text transcript into the prompt. VisionMode.OCR_TEXT is recorded. This works well for text-heavy screens even with text-only models.
  3. Text only (NOT_ATTACHED) — No screenshot. The model sees only the source code. VisionMode.NOT_ATTACHED.

The VisionMode enum lives in ui_analyzer/models/ui_file.py. The pipeline is implemented in UIAnalyzer._run_analysis():

class VisionMode(Enum):
    NOT_ATTACHED = "not_attached"
    MULTIMODAL   = "multimodal"   # model receives raw image bytes
    OCR_TEXT     = "ocr_text"     # text-only model, OCR transcript injected
# Attach screenshot to the user turn when present
vision_mode = VisionMode.NOT_ATTACHED
if screenshot_data:
    b64 = base64.b64encode(screenshot_data).decode()
    multimodal_msg = {
        "role": "user",
        "content": prompt + "\n\nA screenshot of this UI is attached.",
        "images": [b64],
    }
    session.messages.append(multimodal_msg)
    vision_mode = VisionMode.MULTIMODAL
else:
    session.add_user(prompt)

# ... stream the response, then on HTTP 400 "images" error:
if screenshot_data and vision_mode == VisionMode.MULTIMODAL and (
    "400" in str(exc) or "images" in str(exc).lower()
):
    session.messages = [m for m in session.messages if m.get("images") is None]
    ocr_suffix = _build_ocr_suffix(screenshot_data)
    session.add_user(prompt + ocr_suffix)
    vision_mode = VisionMode.OCR_TEXT
    # retry stream ...
The fallback is automatic and transparent to the user. A notification announces the result ("Analysis complete") regardless of which strategy was used. The vision_mode field on UIAnalysis records which path was taken for debugging.

Screenshot attachment

A screenshot can be attached to any source file. Once attached, it travels with the file through analysis and validation. The binding is persisted across sessions in a JSON file in the platform user-data directory.

Naming convention

Attachments are stored at the root of the scanned folder with the name sourcefile.context.png. For example, attaching a screenshot to LoginView.swift creates LoginView.swift.context.png in the same folder. This makes the attachment visible in the sidebar as a scanner item and keeps it co-located with the source.

AttachmentStore

ui_analyzer/services/attachment_store.py owns the persistence layer. The binding is a JSON map from relative_path to attachment_filename, written to a per-user data directory via platformdirs:

import platformdirs

def _store_path() -> Path:
    return Path(platformdirs.user_data_dir("UIAnalyzer")) / "attachments.json"
# Windows : %LOCALAPPDATA%\UIAnalyzer\attachments.json
# macOS   : ~/Library/Application Support/UIAnalyzer/attachments.json

Key functions:

  • attachment_filename(source_name) — returns the .context.png filename for a given source file name.
  • looks_like_attachment(name) — returns True if a filename ends with .context.png. Used by the file scanner to exclude attachment files from the UI file list.
  • set_attachment(attachment_name, for_relative_path) — binds or clears a binding. Pass None as attachment_name to clear.
  • lookup_attachment(relative_path) — looks up the attachment filename for a source file's relative path, or returns None.

UIFile integration

UIFile.attached_image_path holds the relative path of the attachment from the scanned folder root. UIFile.screenshot_abs_path is a property that resolves it to an absolute path:

@property
def screenshot_abs_path(self) -> Optional[Path]:
    if self.attached_image_path and self.folder_path:
        return self.folder_path / self.attached_image_path
    return None

UIAnalyzer.analyze() calls screenshot_abs_path to load the image bytes before building the prompt. If the path is None or the file does not exist, analysis proceeds without any image.

Validate against screenshot

Validation cross-checks a prior text analysis against a real screenshot of the same screen. It asks the model to review what it said and categorise each claim as confirmed, retracted, or newly observed. This catches cases where the model made an incorrect inference from source code alone.

How it works

  1. DetailPanel calls UIAnalyzer.validate_against_screenshot(file, prior_analysis) on a background thread.
  2. The method strips HTML tags from the prior analysis (if HTML mode was used), then calls _build_validation_prompt to produce a prompt containing the prior analysis text.
  3. The prompt and the screenshot bytes are sent to the model via _run_validation(), which tries multimodal first and falls back to OCR text if the model rejects the image.
  4. The model returns a JSON object. _parse_validation_json() parses it into a ValidationResult.

ValidationResult data model

@dataclass
class ValidationClaim:
    text: str

@dataclass
class ValidationRetraction:
    text: str
    original_index: Optional[int] = None  # 1-based paragraph number

@dataclass
class ValidationResult:
    stands_by: list[ValidationClaim]    # claims confirmed by the screenshot
    retracts:  list[ValidationRetraction]  # claims contradicted or misleading
    additions: list[ValidationClaim]    # new facts only visible in the screenshot

The validation prompt and system instructions

The model is given strict output rules via _VALIDATION_SYSTEM_PROMPT:

_VALIDATION_SYSTEM_PROMPT = """\
You are a UI accessibility and visual design expert.
...
Return ONLY the JSON object:
{
  "stands_by":  ["claim still true", ...],
  "retracts":   ["claim that was wrong or misleading", ...],
  "additions":  ["new observation only visible in the screenshot", ...]
}
"""

JSON parsing and fault tolerance

_parse_validation_json(raw) handles malformed responses gracefully:

  • Strips Markdown fences (```json ... ```) that models sometimes add despite the instructions.
  • Falls back to regex extraction of the first {...} block if standard JSON parsing fails.
  • Returns an empty ValidationResult if all parsing fails, so the UI always gets a valid object.

ValidationResultDialog

The result is presented in ValidationResultDialog inside detail_panel.py. Key accessibility properties:

  • Three labeled sections as wx.StaticText headings: Stands By, Retracts, Additions. Each section heading uses SetFont(bold) and SetName() so NVDA/JAWS announce it as a heading-like separator.
  • Each claim is a separate wx.StaticText item so screen readers read each bullet individually.
  • A Copy button and Ctrl C shortcut copy the full result as formatted plain text via _build_plain_text().
  • A Close button and Escape dismiss the dialog and return focus to the detail panel.

Framework API reference injection

Before building any analysis prompt, UIAnalyzer._build_prompt() calls _framework_api_ref(content, file_type) to choose one of eleven framework-specific accessibility reference strings. The chosen reference is injected into the prompt so the model suggests concrete, API-specific fixes rather than generic WCAG advice.

Detection logic

Detection is content-first: the function reads the file's import declarations before looking at the extension. This ensures a .py file importing tkinter gets the tkinter reference (with its limited screen-reader-support caveat), and a .xaml file with the WPF XML namespace gets the WPF reference rather than a MAUI or Avalonia one.

def _framework_api_ref(content: str, file_type: UIFileType) -> str:
    if "import SwiftUI" in content:                       return _SWIFTUI_APIS
    if "react-native" in content:                         return _REACT_NATIVE_APIS
    if "from 'react'" in content or "import React":      return _REACT_APIS
    if "using System.Windows.Forms" in content:           return _WINFORMS_APIS
    if "schemas.microsoft.com/winfx/2006/xaml":          return _WPF_APIS
    if "schemas.microsoft.com/dotnet/2021/maui":         return _MAUI_APIS
    if "github.com/avaloniaui" in content:                return _AVALONIA_APIS
    if "from PyQt6" in content or "from PySide6":        return _PYQT_APIS
    if "import tkinter" in content:                       return _TKINTER_APIS
    if "import wx" in content or "from wx":              return _WXPYTHON_APIS
    # fall back to file_type enum ...
    return ""

The available reference strings are defined as module-level constants in ui_analyzer/services/ui_analyzer.py:

Automated contrast advisor

ui_analyzer/services/contrast_advisor.py runs before the model sees the file. It extracts every #RRGGBB and #RGB hex color from the source text, computes the WCAG relative luminance for each, and compares every pair to find contrast failures.

The function contrast_report(content: str) -> str returns a human-readable block (or an empty string if all colors pass). It is injected into the analysis prompt so the model can reference exact ratios and suggest specific fixes:

cr = contrast_report(content)
if cr:
    extra += cr + "\n\n"

The report format the model receives looks like:

COLOR CONTRAST REPORT (auto-extracted from source):
• #afafaf on #ffffff — ratio 2.22:1 (FAILS AA normal text 4.5:1)
  → suggested fix: #767676 (ratio 4.61:1)
• #0071e3 on #ffffff — ratio 3.62:1 (FAILS AA normal text; PASSES AA large text)
  → suggested fix: #0059b3 (ratio 6.54:1)

The model is instructed to cite these ratios verbatim in the Accessibility section of its output so the user gets precise numbers, not guesses.

Session management and context rolling

Each file has its own FileSession object, stored in UIAnalyzer._sessions[file_id]. A session holds the full message history (system + user + assistant turns) so follow-up questions are aware of everything that was said earlier.

FileSession

class FileSession:
    def __init__(self, file_id, file_name, file_type,
                 content_prefix, mode, table_format):
        self.messages: list[dict] = []   # full chat history
        self.chars_exchanged = 0         # running total to trigger roll
        self.content_prefix = content_prefix  # first 2,000 chars for re-anchoring
        self.roll_count = 0

Context rolling

Local models have a finite context window. When chars_exchanged exceeds CONTEXT_BUDGET = 12_000, the follow-up handler calls _roll_context(file_id) before sending the next question:

CONTEXT_BUDGET = 12_000  # chars before rolling

def _roll_context(self, file_id: str) -> None:
    s = self._sessions[file_id]
    # 1. Ask the model to summarise the conversation so far
    summary_prompt = (
        "Summarise the key facts from our conversation about this UI in 4–6 sentences: "
        "screen name/purpose, UI elements and labels, colours (name + hex), layout, "
        "professional quality verdict, accessibility notes, and follow-up findings."
    )
    s.add_user(summary_prompt)
    summary = self._client.respond(self.model, s.messages)

    # 2. Replace the full history with: system + summary + file opening lines
    sys_instructions += f"\n\nCONTEXT FROM EARLIER:\n{summary}\nFile: {s.file_name}"
    s.messages = [
        {"role": "system",    "content": sys_instructions},
        {"role": "user",      "content": f"Continue our discussion...{s.content_prefix}"},
        {"role": "assistant", "content": "Understood — continuing from where we left off."},
    ]
    s.chars_exchanged = len(summary)
    s.roll_count += 1

Rolling is transparent to the user: the conversation continues without interruption, and the compressed history keeps the model anchored to the original file content and earlier findings.

Design specification detection

Markdown files that are clearly design specifications get a different analysis prompt than ordinary Markdown documentation. The detection function is:

def _is_design_spec(file_name: str, file_type: UIFileType) -> bool:
    if file_type != UIFileType.MARKDOWN:
        return False
    skip = ("readme", "changelog", "contributing", "license", "history", "authors")
    return not any(file_name.lower().startswith(s) for s in skip)

Any Markdown file whose name does not start with one of the standard documentation keywords is treated as a design spec and routed to _design_spec_prompt, which asks the model to cover:

  1. Color System — Every color defined with its name, hex, intended use, and WCAG contrast check.
  2. Typography — Every type style with family, weight, size, line-height, and usage.
  3. Spacing, Layout, and Grid — Spacing scale, grid system, breakpoints, and layout tokens.
  4. Components and Patterns — Every component with states (default, hover, active, disabled, error) and usage rules.
  5. Accessibility Design Decisions — Focus styles, contrast choices, touch targets, motion preferences, screen reader guidance.

This prompt is used for files like design.md, tokens.md, or style-guide.md where the content is design token definitions rather than prose documentation.

Drag and drop

One drop target is registered, on the sidebar:

  • SidebarPanel — wx.FileDropTarget subclass FolderDropTarget. Accepts any dropped folder and calls on_drop_folder with the Path. Wired up in MainFrame to call _load_folder, which kicks off a background scan.

The drop target only accepts directories (not individual files) because the rest of the app is folder-centric — dropping a single file would have no natural place to go.

Prompt routing

UIAnalyzer.analyze(file, mode, fmt, on_chunk) routes to one of five prompt templates based on file type, file name, and user-selected mode. The routing is checked in this order:

Prompt routing decisions — checked top to bottom
Priority Condition Prompt template Output
1 file.file_type.is_image _image_prompt Visual Appearance, Layout, Text Content, Design Quality, Accessibility
2 file type is Markdown AND not README/CHANGELOG/etc. _design_spec_prompt Color System, Typography, Spacing & Layout, Components & Patterns, Accessibility Design Decisions
3 mode == TABLE, format == HTML _html_table_prompt TL;DR <details> block + accessible HTML5 <table>
4 mode == TABLE, format == MARKDOWN _markdown_table_prompt Summary paragraph + Markdown table
5 Any other type, mode == PROSE _prose_prompt Visual Appearance, Ease of Use, Professional Quality, Accessibility, Other Details

Each prompt template is a Python function returning a multi-line string. They are designed to be edited in one place; if you change a section, change it consistently across all five.

Accessibility implementation

The app is built WCAG 2.2 AA accessible end to end. The implementation targets NVDA, JAWS, Narrator, and VoiceOver simultaneously. The patterns below are the ones that matter.

SetName + SetToolTip on every interactive control

wx exposes a widget's accessible name through the SetName(...) method. Screen readers (NVDA, JAWS, Narrator, VoiceOver) announce this name when the widget receives focus. SetToolTip(...) provides a longer description on hover/focus.

Every wx.Button, wx.TextCtrl, wx.Choice, wx.ListCtrl, and modal dialog in the app has both. The label-only StaticText next to a field acts as the visible label; SetName on the control makes the screen reader announce the same thing.

44 × 44 minimum touch / click target

WCAG 2.5.5 requires interactive targets to be at least 44 × 44 CSS pixels. Every button in the app uses size=(W, 44) (or taller) to meet this. The toolbar's "Open Folder" button, the model picker's "Refresh model list" button, and all the detail-panel action buttons (Copy, Save, Analyze, Ask) are 44 px tall.

Paired label + control

The follow-up question text field has a StaticText("Follow-up:") next to it, and the TextCtrl is given SetName("Follow-up question"). Screen readers announce "Follow-up question, edit, blank" when focus enters the field.

Show / hide via state, not via Enable / Disable when possible

The detail panel's five state panels (empty / ready / loading / result / error) are stacked in the same sizer; _show_state(name) calls Show(name == state) on each. The follow-up bar and the copy/save buttons are also shown or hidden rather than enabled / disabled when the state changes, because hiding removes the control from the tab order and screen-reader cursor more predictably than the disabled state.

The exception is the Table format dropdown: it is Disable()-d (not hidden) when the user picks Prose mode. Hiding it would silently remove a control the screen reader already enumerated, which is more disorienting than disabling.

Focus management after analysis

When an analysis finishes, MainFrame._on_analysis_done calls self._detail._result_view.SetFocus(). This moves focus to the WebView showing the analysis, so the user's screen reader immediately begins reading the result. Without this, focus would stay on the Analyze button and the user would have to Tab through several controls to reach the output.

Live region status updates

The status bar at the bottom of the window is the app's live region. Every state transition (scan started, analysis started, analysis complete, error) calls self._status.SetStatusText(...). The status bar fires UIA / IAccessible2 events that NVDA, JAWS, Narrator, and VoiceOver all monitor — they announce the new text automatically.

Accessible WebView for analysis output

The wx_accessible_webview library (an AccessibleWebView wrapper around wx.html2.WebView) exposes the rendered HTML through IAccessible2 / UIA so screen readers can navigate headings, tables, and form controls inside the WebView. The fallback is the plain wx.html2.WebView (still readable, just less rich in the AT tree).

The HTML analysis output uses:

  • <h2>Question</h2> / <h2>Answer</h2> for project Q&A — lets the screen reader's heading rotor jump between Q&A pairs.
  • <details class="tldr" open> for the HTML table TL;DR — announced as a collapsible region with the summary text.
  • <table> with <thead>, <tbody>, <th scope="col"> — lets the user navigate cell by cell with the screen reader's table commands.

Color contrast in the chrome

The default wx theme meets WCAG AA on Windows (System theme) and macOS. The "Open Folder" button, the toolbar labels, and the status bar text all use the system foreground / background colors, so they inherit the high-contrast settings when the user enables High Contrast Mode in Windows or Increase Contrast in macOS.

Custom app icon

The bundled assets/app_icon.png is set on the wx.App via app.SetIcon(...) and on the MainFrame via frame.SetIcon(...) in main.py. On Windows, this shows the icon in the window title bar, the Alt-Tab switcher, and the taskbar. On macOS, the title bar only (the Dock icon is set by the .app bundle's Info.plist, which doesn't exist in python main.py dev runs).

Keyboard shortcuts

All shortcuts are defined in MainFrame._build_accelerators as wx.AcceleratorEntry records. They work regardless of which child control has focus, so they are available from inside the follow-up text field, the WebView, and the model picker dialog.

Known gotchas

  • wx is single-threaded: every SetStatusText, SetItem, SetLabel, and SetFocus call from a worker thread must be wrapped in wx.CallAfter. Forgetting this works on a lucky run, then crashes randomly inside stream_chunk.
  • Ollama wraps HTML in Markdown fences: even with explicit "do not use Markdown" instructions, the model sometimes wraps the HTML output in ```html ... ```. _extract_html(raw) finds the first <!DOCTYPE or <html substring and strips everything before it, then strips a trailing ``` if present. Call it on every HTML response.
  • HTML mode requires a complete document before render: the AccessibleWebView parses the whole HTML at once. During streaming, DetailPanel uses a plain wx.TextCtrl for the streaming output and switches to the WebView only when the response is complete.
  • FileDialog wildcard for HTML saves: on Windows, the HTML files (*.html)|*.html wildcard requires the full *.html pattern, not just *.html. Getting this wrong silently saves the file with no extension on some Windows builds.
  • SetName (not SetLabel) drives screen reader name: SetLabel changes the visible text. SetName changes the accessible name. NVDA / JAWS / Narrator / VoiceOver all use the accessible name. If you only set the label, the screen reader announces the type ("button") with no name.
  • 44 px button height: the default wx.Button is too short to meet WCAG 2.5.5. Always pass size=(W, 44) or use a wrapper Panel with SetMinSize((-1, 44)).
  • Ollama not running: GET /api/tags returns a connection error. The startup flow in MainFrame._auto_select_model catches this and shows a wx.MessageBox with a clear "how to start" message. The window still opens so the user can read the message with their screen reader.
  • Tesseract not on PATH: pytesseract.image_to_string raises TesseractNotFoundError. The image analysis path catches this and shows a clear error in the detail panel.
  • Refresh model list after pulling: the model picker caches its list at construction time. After ollama pull, the user must click the Refresh model list button to see the new model.
  • AccessibleWebView import is optional: if wx_accessible_webview is not installed, DetailPanel falls back to wx.html2.WebView silently. The fallback works but exposes less of the rendered HTML to screen readers. To get the full a11y experience on Windows, install pip install wx-accessible-webview.

Run instructions

First time setup

cd "path/to/UI-Analyzer"
python3 -m venv .venv
source .venv/bin/activate          # Windows: .venv\Scripts\activate
pip install -r requirements.txt
python main.py

Verify Ollama is running

ollama list                            # should print at least one model
ollama pull qwen2.5-coder:7b           # if empty
ollama serve                           # if not running

Verify the app icon loads

python3 -c "import wx; app=wx.App(); from pathlib import Path; \
  print(wx.Image(str(Path('assets/app_icon.png')), wx.BITMAP_TYPE_PNG).IsOk())"

Should print True. If it prints False, the PNG is corrupt or missing.

Verify the package imports

python3 -c "from ui_analyzer.services.ui_analyzer import UIAnalyzer; \
  from ui_analyzer.views.main_frame import MainFrame; print('OK')"

Should print OK. If it raises ImportError, run pip install -r requirements.txt from inside the activated venv.

Headless smoke test (CI / server)

python3 -c "import wx; app=wx.App(); \
  from ui_analyzer.views.main_frame import MainFrame; \
  f = MainFrame(); f.Show(); print('Window constructed')"

Constructs the window without entering the main loop. Useful in CI to catch widget-construction errors.

Build a Windows .exe (optional)

pip install pyinstaller
pyinstaller --onefile --windowed \
  --add-data "assets;assets" \
  --name "UI Analyzer" \
  main.py

The resulting dist/UI Analyzer.exe includes the icon. PyInstaller will use the .png icon for the window title bar; for a real .exe file icon (shown in File Explorer), generate an .ico from the PNG and pass --icon=app_icon.ico.
