What UI Analyzer does

UI Analyzer reads your app's source code files and describes what the user interface looks like to sighted people. It gives you the information you need to make confident decisions about your app's visual design, layout, colors, and accessibility — even if you cannot see the screen.

It uses a local AI model served by Ollama. Your code never leaves your machine. No account, no subscription, and no internet connection is needed for analysis. Choose any model you have pulled locally — from the tiny 1 GB qwen2.5-coder:1.5b for quick checks to the full 7 GB qwen2.5-coder:7b for the deepest analysis.

UI Analyzer tells you:

  • What colors are used and what they look like
  • How the layout is arranged and what the spacing looks like to a sighted person
  • What fonts and typography are used
  • Whether the interface looks professional and polished
  • Whether there are potential accessibility problems such as low color contrast

Every interactive control has an accessible name, every button meets the 44 × 44 minimum touch target, every dialog can be operated with the keyboard alone, and every screen reader (NVDA, JAWS, Narrator, VoiceOver) can read every screen. The app is built WCAG 2.2 AA accessible end to end.

Requirements

  • Windows 10 or 11 (64-bit). macOS 14+ is also supported for development.
  • Python 3.10 or later — python.org
  • Ollama installed and running — ollama.com/download
  • At least one model pulled locally — for example ollama pull qwen2.5-coder:7b
  • (Optional) Tesseract OCR for screenshot analysis — see the Screenshots section
  • (Windows only) Edge WebView2 Runtime — ships with Windows 11; installer at Microsoft Edge WebView2

To verify Ollama is running, open a terminal and run ollama list. You should see at least one model name. If the list is empty, pull one first: ollama pull qwen2.5-coder:7b.

If Ollama is not running when you start UI Analyzer, the app shows a clear message telling you how to start it and which model to install. The window still opens so you can read the message with your screen reader.

Getting started

  1. Open UI Analyzer. If you are on Windows, the bundled app icon appears in the title bar and on the taskbar.
  2. Press Ctrl O to open a folder dialog, or drag a folder from File Explorer and drop it onto the sidebar.
  3. The sidebar on the left lists all supported UI files found in that folder. Use Up and Down to move through the list; NVDA / JAWS / Narrator / VoiceOver will read each file's name, type, and status (Ready / Analyzing / Analyzed / Error).
  4. Select a file in the sidebar and press Enter, or click the Analyze button (or press Ctrl R).
  5. The local model streams a description in real time. The output view announces "Analysis in progress" while it streams, then "Analysis complete" when it finishes. Focus moves to the output view so your screen reader begins reading the result.
  6. Read the output, copy it, or save it as a file.

Drag and drop from File Explorer / Finder

You can drag a folder from File Explorer (Windows) or Finder (macOS) and drop it anywhere on the sidebar. The folder becomes the active project folder and is scanned immediately.

Keyboard-first workflow

Every action is reachable with the keyboard. The most common shortcuts are Ctrl O (open folder), Ctrl R (analyze / re-analyze), Ctrl M (choose AI model), Ctrl Shift A (build project context), Ctrl Shift I (ask project question), Ctrl Shift C (copy output), and Ctrl S (save output). See the Keyboard shortcuts section for the full list.

Understanding the output

Prose mode (default)

A full written description in five sections:

  1. Visual Appearance — Colors (name and hex code), layout, fonts, spacing, icons, and overall composition.
  2. Ease of Use for Sighted Users — Is the interface clear and easy to navigate visually? Are interactive elements obvious?
  3. Professional Quality — Does it look polished? The model gives a direct verdict with reasons.
  4. Accessibility — Color contrast, touch target sizes, keyboard navigation, and accessibility labels.
  5. Other Important Details — Animations, error states, empty states, conditional UI.

Use the Format dropdown at the top of the detail panel to switch between Prose and Table.

Table mode

A structured table of every UI element. Switch to Table mode, then choose either format from the second dropdown:

  • Markdown table — Plain text, good for copying or reading with a screen reader.
  • HTML table — A rendered accessible table with proper <table>, <thead>, <tbody>, <th scope> semantics. NVDA / JAWS / Narrator / VoiceOver all announce it as a table and let you navigate cell by cell. The HTML output also starts with a TL;DR <details> block — a 3–5 sentence summary you can read first before expanding into the detailed sections.

Design specification files

Markdown files (such as design.md) are automatically recognized as design specs and produce a design-focused output with five sections:

  1. Color System
  2. Typography
  3. Spacing, Layout, and Grid
  4. Components and Patterns
  5. Accessibility Design Decisions

Framework-specific accessibility suggestions

UI Analyzer detects which UI framework your file uses and injects a relevant accessibility API reference into the analysis prompt, so suggestions are specific and actionable. The set of supported frameworks covers the full cross-platform landscape:

  • SwiftUI — .accessibilityLabel, @AccessibilityFocusState, touch targets, AccessibilityNotification, and more
  • React / React Native — ARIA attributes, accessibilityLabel, accessibilityRole, focus management with useRef
  • Vue / Svelte — :aria-label, aria-live, router announcements, :focus-visible
  • HTML / CSS — Semantic elements, <dialog>, <details>, ARIA, dark mode, reduced motion
  • PyQt6 / PySide6 / tkinter / wxPython — setAccessibleName, setFocusPolicy, SetName / SetToolTip, keyboard bindings
  • WinForms / WPF / .NET MAUI / Avalonia — AccessibleName, AutomationProperties.Name, SemanticProperties.Description, KeyboardNavigation.TabIndex

Automated color contrast check

UI Analyzer extracts every hex color from your source file, computes the exact WCAG contrast ratio against white, black, and dark backgrounds, and flags anything that fails the AA standard (4.5:1 for normal text, 3:1 for large text and UI components). Where a color fails, it also suggests an alternative color — with the same hue — that just passes. These exact ratios and suggestions appear in the Accessibility section of all three output formats.

Screenshots and images

UI Analyzer can analyze screenshots and other images of user interfaces. Supported formats include PNG, JPG, HEIC, WEBP, TIFF, BMP, and GIF (up to 10 MB).

To add an image, open a folder that contains image files — they appear automatically in the sidebar.

When you analyze an image, UI Analyzer:

  1. Detects the image dimensions and identifies the likely device type (such as iPhone or iPad based on screen proportions).
  2. Reads all visible text using Tesseract OCR (Windows) or the built-in OCR, noting where each piece of text appears on screen (for example: "top-left", "center", "bottom-right").
  3. Passes this information to the local model to generate a complete visual description.

Windows — install Tesseract: download the installer from github.com/UB-Mannheim/tesseract/wiki and make sure the tesseract command is on your PATH.

macOS — install Tesseract: brew install tesseract

Image analysis is text-based. For best results with images that have very little text, add a descriptive title or question in the follow-up field after analysis to ask about specific visual elements.

Asking follow-up questions

After an analysis finishes, a text field labeled Follow-up question appears at the bottom of the detail panel. Use Tab to move focus to it, type your question, and press Enter (or click the Ask button).

Example questions:

  • "What is the exact hex color of the primary button?"
  • "Does the text contrast meet WCAG AA standards?"
  • "How large is the navigation bar in points?"
  • "What happens when the user taps the back button?"
  • "Is this image showing an iPhone or iPad?"

The model remembers the full conversation for each file, so you do not need to repeat context between questions. Follow-up answers are emitted as HTML fragments with <h2>Question</h2> / <h2>Answer</h2> headings, so you can use NVDA / JAWS / Narrator / VoiceOver heading navigation to jump between Question and Answer blocks and between individual list items.

Build Project Context

Press Ctrl Shift A or click Build Project Context at the bottom of the sidebar to analyze all files in your project folder at once. This creates a summary of every screen so you can ask cross-project questions:

  • "Is the color consistent across all screens?"
  • "Which screens have accessibility problems?"
  • "Does the typography look the same throughout the app?"

The status bar shows the build progress in real time (e.g. "Building context: 3/8…") and your screen reader announces the live count. Once the build is complete, press Ctrl Shift I to open the project question dialog and ask anything about the project as a whole.

Choosing an AI model

UI Analyzer works with any model you have pulled into Ollama. Open the File menu and choose Choose AI Model…, or press Ctrl M. The model picker shows every installed model with a short description of what it is best for, minimum RAM, and on-disk size. Models marked (recommended) are the best fit for the file type currently selected in the sidebar.

Suggested models:

  • qwen2.5-coder:7b — best balance of speed and quality for most analysis. Needs about 8 GB of RAM.
  • qwen2.5-coder:3b — faster, lower memory, slightly less detailed. Good on 8 GB laptops.
  • llama3.1:8b — strong general model, slightly slower than the Qwen coders.
  • qwen2.5-coder:1.5b — very fast, very small. Useful for quick checks when you don't need depth.

To install a model, open a terminal and run, for example, ollama pull qwen2.5-coder:7b. Then click Refresh model list in the model picker to see it appear.

If you change the model while an analysis is in progress, the new model takes effect on the next file. In-flight analyses finish on the old model so nothing is lost.

Saving and copying

  • Copy (Ctrl Shift C) — Copies the full analysis text to the clipboard. The status bar confirms "Copied to clipboard" and your screen reader announces it.
  • Save (Ctrl S) — Opens a Save dialog. The default extension is .html for HTML output and .md for Markdown / Prose output. Choose the format from the file-type dropdown in the dialog.

Saved HTML files are fully self-contained — they include the styles for both light and dark mode, and they work in any modern browser. The <table> structure is preserved, so you can navigate them with the same screen reader you used to read them in the app.
