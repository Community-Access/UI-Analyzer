What UI Analyzer does

UI Analyzer reads your app's source code files and describes what the user interface looks like to sighted people. It gives you the information you need to make confident decisions about your app's visual design, layout, colors, and accessibility — even if you cannot see the screen yourself.

It uses a local AI model served by Ollama. Your code never leaves your machine. No account, no subscription, and no internet connection is needed for analysis. You choose any model you have downloaded locally — from a small, fast 1 GB model for quick checks, up to a full 7 GB model for the deepest analysis.

UI Analyzer tells you:

  • What colors are used and what they look like to sighted users
  • How the layout is arranged and how the spacing feels
  • What fonts and typography are used
  • Whether the interface looks professional and polished
  • Whether there are potential accessibility problems such as low color contrast or missing keyboard support
  • Whether your code matches what a screenshot shows — using the Validate feature

Every interactive control has an accessible name, every button meets the minimum 44×44 touch target size, every dialog can be operated with the keyboard alone, and every screen reader (NVDA, JAWS, Narrator, VoiceOver) can read every screen. The app is built to WCAG 2.2 AA standards end to end.

Requirements

  • Windows 10 or 11 (64-bit), or macOS 14 or later
  • Python 3.10 or later — python.org/downloads
  • Ollama installed and running — ollama.com/download
  • At least one model pulled locally — for example: ollama pull qwen2.5-coder:7b
  • (Optional) Tesseract OCR for screenshot text reading — see the Screenshots section
  • (Windows only) Edge WebView2 Runtime — ships automatically with Windows 11; installer available at Microsoft's Edge WebView2 page for Windows 10

To check that Ollama is running, open a terminal and type ollama list. You should see at least one model name. If the list is empty, pull a model first: ollama pull qwen2.5-coder:7b.

Note: If Ollama is not running when you start UI Analyzer, the app shows a clear message telling you how to start it and which model to install. The window still opens so you can read the message with your screen reader.

Getting started

  1. Open UI Analyzer. If you are on Windows, the app icon appears in the title bar and on the taskbar.
  2. Press Ctrl O (Windows) or Cmd O (macOS) to open a folder dialog, or drag a folder from File Explorer or Finder and drop it onto the sidebar.
  3. The sidebar on the left lists all supported UI files found in that folder. Use the Up and Down arrow keys to move through the list. Your screen reader announces each file's name, type, and status (Ready, Analyzing, Analyzed, or Error).
  4. Select a file and press Enter, or press Ctrl R / Cmd R to analyze it.
  5. The AI model streams a description in real time. While streaming, your screen reader announces "Analysis in progress". When the analysis is complete, it announces "Analysis complete" and focus moves to the output view so you can begin reading.
  6. Read the output, copy it with Ctrl Shift C, or save it with Ctrl S.

Keyboard-first workflow

Every action is reachable with the keyboard. The most common shortcuts: Ctrl O to open a folder, Ctrl R to analyze, Ctrl Shift V to validate against a screenshot, Ctrl Shift A to build project context, Ctrl Shift I to ask a project question, Ctrl Shift C to copy output, and Ctrl S to save. On macOS, replace Ctrl with Cmd. See the full Keyboard shortcuts table.

Drag and drop

You can drag any folder from File Explorer (Windows) or Finder (macOS) and drop it anywhere on the sidebar. UI Analyzer immediately scans the folder and lists the files it finds.

Command Line Interface (CLI)

For power users and automation, UI Analyzer includes a standalone CLI tool. This allows you to analyze files without opening the GUI, making it ideal for scripts, CI/CD pipelines, or AI-driven workflows.

Basic Usage

Run the tool using Python from your terminal:

python ui_analyzer.py path/to/file.py [options]

Common Options

  • --backend [auto|ollama|openai|claude] — Choose the AI engine. Defaults to auto.
  • --output-mode [prose|table|html] — Change the analysis format.
  • --output FILE — Save the result to a file instead of printing to the terminal.
  • --spec FILE — Check the UI against a Markdown visual standards specification.

Working with AI Agents

If you are using an AI agent (like Hermes) to help you develop your app, the agent can use this CLI to "see" your UI. The agent can run the tool on your source files, read the prose or table output, and then suggest specific accessibility or design improvements directly in your code.

Tip: Agents can pipe the output to a file or use the --output-mode html to generate a report they can then analyze for you.

Output formats

Prose mode (default)

A full written description in six sections, each with a navigable heading so you can jump between them with your screen reader's heading navigation:

  1. Visual Appearance — Colors (name and hex code), layout, fonts, spacing, icons, and overall composition described in plain language.
  2. Ease of Use for Sighted Users — Is the interface clear to navigate visually? Are interactive elements obvious? Is there enough visual feedback?
  3. Professional Quality — Does it look polished? The model gives a direct verdict (Polished / Mostly Polished / Needs Polish) with specific reasons.
  4. Accessibility — Color contrast ratios, touch target sizes, keyboard navigation, and presence of accessibility labels. The model suggests specific code fixes for your framework.
  5. Other Important Details — Animations, error states, empty states, conditional UI, and platform-specific behaviors.
  6. Screen Layout — A region-by-region description from top to bottom, naming each zone and what a sighted user sees in it.

Use the Format dropdown at the top of the detail panel to switch between Prose and Table.

Table mode

A structured table of every UI element, covering element type, label text, color (name + hex), position, whether it is interactive, spacing/size, and notes. Choose either sub-format:

  • Markdown table — Plain text. Good for copying into documentation or reading with a screen reader in a plain text editor.
  • HTML table — A rendered accessible table with correct <table>, <thead>, <th scope> semantics. NVDA, JAWS, Narrator, and VoiceOver all announce it as a proper table so you can navigate cell by cell with arrow keys. The HTML output starts with a TL;DR summary block — 3 to 5 sentences giving the headline findings — before the full tables.

Design specification files

Markdown files that are clearly design specifications (not README or changelog files) automatically receive a design-focused analysis with five sections: Color System, Typography, Spacing & Layout, Components & Patterns, and Accessibility Design Decisions. This is useful for reviewing a design token file or a design system document.

Framework-specific accessibility suggestions

UI Analyzer detects which UI framework your file uses and injects a relevant accessibility API reference into the analysis prompt. This means suggestions are specific and actionable, not generic. Supported frameworks include:

  • Python / Desktop — SetName, SetToolTip, setAccessibleName, keyboard bindings
    • Slightly modified a line here to be more general if needed.
  • React / React Native — ARIA attributes, accessibilityLabel, accessibilityRole, useRef focus management, aria-live
  • Vue / Svelte — :aria-label, aria-live, :focus-visible, reduced-motion
  • HTML / CSS — Semantic elements, <dialog>, ARIA, dark mode, reduced motion
  • wxPython / PyQt6 / PySide6 / tkinter — SetName, SetToolTip, setAccessibleName, keyboard bindings
  • WinForms / WPF / .NET MAUI / Avalonia — AccessibleName, AutomationProperties.Name, SemanticProperties.Description

Automated color contrast check

Before sending a file to the AI, UI Analyzer extracts every hex color from your source file and computes the WCAG contrast ratio against white, black, and dark backgrounds. Any color that fails the AA standard (4.5:1 for normal text, 3:1 for large text and UI components) is flagged with a suggested replacement — same hue, adjusted lightness — that just passes. These ratios and suggestions appear in the Accessibility section of all output formats.

Analyzing screenshots and images

UI Analyzer can analyze screenshots of user interfaces as standalone files. Supported formats: PNG, JPG, HEIC, WEBP, TIFF, BMP, and GIF (up to 10 MB). Drop a folder containing image files and they appear in the sidebar automatically.

When you analyze an image, UI Analyzer:

  1. Detects the image dimensions and notes the likely device type (for example, a 390×844 image is likely an iPhone 14).
  2. Reads all visible text using Tesseract OCR, noting where each piece of text appears on screen (for example: "top-left", "center", "bottom-right").
  3. Passes the image and text layout to the local AI model to generate a complete visual description in the same six-section format as source file analysis.

Installing Tesseract (optional but recommended)

  • Windows: download the installer from the UB Mannheim Tesseract GitHub page and make sure the tesseract command is on your PATH.
  • macOS: run brew install tesseract in Terminal.
Note: If Tesseract is not installed, UI Analyzer still works but image analysis is less accurate. The model will infer layout from any text it can see in the image bytes (if using a vision-capable model), but won't have precise text positions.

Attaching a screenshot to a source file

You can attach a screenshot to any source code file in the sidebar. Once attached, the screenshot is included with every analysis of that file — the AI model can see or read both the code and the visual output at the same time. This is useful for catching discrepancies between what the code says and what the screen actually shows.

How to attach a screenshot

  1. In the sidebar, right-click (or press the context menu key) on any source file.
  2. Choose Attach screenshot… from the menu.
  3. Select an image file from the dialog. The image is copied to your project folder with the name filename.context.png (for example, LoginView.swift.context.png).
  4. The sidebar item shows a camera icon to indicate that a screenshot is attached.
  5. Run the analysis as normal with Ctrl R. The model now receives both the source code and the screenshot.

How attachment affects analysis

If the model you are using supports vision (multimodal) input, the image is sent directly alongside the source code as a binary attachment. If the model only supports text, UI Analyzer automatically falls back to OCR text extraction from the screenshot and injects the text transcript into the prompt instead. Either way, you get the best analysis available for the model you have chosen.

Removing a screenshot attachment

Right-click the file in the sidebar and choose Remove attachment. The screenshot file is removed from the project folder and the binding is cleared. Future analyses of the file will use the source code only.

Tip: Attaching a screenshot is especially powerful before running Validate. The AI can compare the analysis it produced from the code against the actual rendered screen to catch things that look different than expected.

Validate against screenshot

After analyzing a source code file that has a screenshot attached, you can ask UI Analyzer to validate its analysis. Validation sends the previous analysis text and the attached screenshot to the AI model at the same time, and asks the model to review what it said in the light of what it can actually see.

Press Ctrl Shift V (Windows) or Cmd Shift V (macOS) to start validation. A notification appears while validation is in progress, and a second notification announces "Validation complete" when it finishes.

What the Validation Result dialog shows

The result appears in an accessible dialog with three sections, each announced as a heading by your screen reader:

  • Stands By — Claims from the original analysis that the screenshot confirms are correct.
  • Retracts — Claims from the original analysis that the screenshot contradicts or that are misleading. Each retraction includes a brief explanation of what was wrong.
  • Additions — New observations that are visible in the screenshot but could not be inferred from the source code alone, such as actual rendered colors, image content, or layout subtleties.

Use your screen reader's heading navigation to jump between the three sections. Every item in each section is read as a separate list item for easy navigation.

The dialog has a Copy button (Ctrl C also works from within the dialog) that copies the full validation result as formatted plain text to the clipboard, and a Close button (or Escape) to dismiss it.

Note: Validation requires a screenshot to be attached to the file. If no screenshot is attached, you will see a message explaining how to attach one. See the Attaching a screenshot section.

Follow-up questions

After an analysis finishes, a text field labeled Follow-up question appears at the bottom of the detail panel. Press Tab to move focus to it, type your question, and press Enter (or click Ask).

The AI model remembers the full conversation for each file, so you do not need to repeat context between questions. Follow-up answers appear in the output area with Question and Answer headings so you can navigate between them with your screen reader's heading key.

Example questions to try:

  • "What is the exact hex color of the primary button?"
  • "Does the text contrast meet WCAG AA standards?"
  • "How large is the navigation bar in points?"
  • "What would a sighted person notice first when they open this screen?"
  • "Is there enough visual difference between the active and inactive tab states?"
  • "What does the empty state look like?"
Tip: You can ask as many follow-up questions as you like. The conversation is automatically compressed in the background if it gets very long, so the model always has enough context to answer accurately without you needing to do anything.

Build Project Context and ask project questions

Press Ctrl Shift A (Windows) or Cmd Shift A (macOS), or click Build Project Context at the bottom of the sidebar. UI Analyzer reads every file in the project folder and generates a short 4-line summary of each screen covering its purpose, colors, visual quality, and any issues found.

The status bar shows the build progress in real time ("Building context: 3 of 8…") and your screen reader announces the live count. Building is quick — typically a few seconds per file.

Once built, press Ctrl Shift I (Windows) or Cmd Shift I (macOS) to open the project question dialog and ask anything about the project as a whole:

  • "Is the color palette consistent across all screens?"
  • "Which screens have accessibility problems?"
  • "Does the typography look the same throughout the app?"
  • "Are the button styles consistent or do they vary between screens?"
  • "What is the overall visual quality of this app?"

Project question answers are formatted with HTML headings so you can navigate between the Question and Answer sections with your screen reader's heading key.

Note: Project Context is built once and reused until you open a different folder or rebuild it. If you add new files to the project, rebuild the context to include them.

Crawl a localhost site

The Localhost Site Crawler lets you point UI Analyzer at a running local web server — for example http://localhost:3000 — and automatically fetch every linked page. Each page is saved as an HTML file and added to the sidebar as a file you can select and analyze. This is useful for auditing a local web application, a static site generator output, or a server-side rendered app without having to open and drag individual files.

How to crawl a site

  1. Start your local web server so it is accepting connections (for example: python -m http.server 3000).
  2. In UI Analyzer, open File → Crawl Localhost Site… or press Ctrl Shift W (Windows) or Cmd Shift W (macOS).
  3. In the dialog that appears, enter the starting URL such as http://localhost:3000.
  4. Optionally adjust the settings:
    • Max pages — how many pages to fetch before stopping (default: 50).
    • Max depth — how many link-levels deep to follow from the start page (default: 3).
    • Pause between requests — a small delay between each page fetch to avoid overloading the server (default: 0.5 seconds).
  5. Click Start Crawl. The dialog closes and the crawl begins in the background.

During the crawl

The status bar at the bottom of the window shows live progress, for example: Crawling: 4/12 pages — http://localhost:3000/about. Your screen reader announces each update automatically.

Press Escape at any time to cancel the crawl. Pages already fetched remain in the sidebar.

After the crawl completes

When the crawl finishes, a notification appears and the status bar confirms completion. The crawled pages are appended to the existing sidebar list — any files you already had open are still there. The first crawled page is selected automatically so you can begin reading or analyzing straight away.

Each page appears in the sidebar with the filename derived from its URL path. You can select any page and analyze it with Ctrl R (Windows) or Cmd R (macOS) just as you would with any other file.

Note: The crawler uses static HTTP fetching. It downloads the raw HTML the server sends, exactly as a browser would receive it before JavaScript runs. This means it works well for static HTML sites, server-side rendered apps (such as those built with Django, Rails, or Astro), and documentation sites. React, Vue, or Angular apps that render entirely in the browser will show the unrendered source HTML rather than the finished page content.
Tip: A static three-page test site is included with the app at tests/fixtures/sample-site/. To try a quick crawl: open a terminal, run python -m http.server 3000 from that folder, then use File → Crawl Localhost Site… and enter http://localhost:3000.

Choosing an AI model

UI Analyzer works with any model you have downloaded into Ollama. Open the File menu and choose Choose AI Model…, or press Ctrl M (Windows) / Cmd M (macOS). The model picker shows every installed model and marks the best fit for the file type you currently have selected.

Your model choice is saved automatically and remembered the next time you open the app. To change it, just open the picker again.

Suggested models to pull:

  • qwen2.5-coder:7b — Best balance of speed and quality for source code analysis. Needs about 8 GB of RAM. Pull it with: ollama pull qwen2.5-coder:7b
  • qwen2.5-coder:3b — Faster and lower memory. Good on 8 GB laptops.
  • llama3.2-vision:11b — Multimodal model that can see screenshots directly. Good for the Validate feature.
  • qwen2.5-coder:1.5b — Very fast and very small. Useful for quick checks.

To install a model, open Terminal and run: ollama pull model-name. Then click Refresh model list in the model picker to see it appear.

Settings

Open File → Settings… to configure the Ollama server URL (useful if Ollama is running on a different port or machine), and to manage API keys if you want to use a cloud AI provider (Anthropic Claude or OpenAI) instead of a local model.

Note: If you change the model while an analysis is in progress, the new model takes effect on the next file. In-flight analyses finish on the previous model so nothing is interrupted or lost.

Saving and copying output

  • Copy (Ctrl Shift C) — Copies the full analysis text to the clipboard. The status bar confirms "Copied to clipboard" and your screen reader announces it.
  • Save (Ctrl S) — Opens a Save dialog. The default extension is .html for HTML Table output and .md for Markdown and Prose output. Choose the format from the file-type dropdown in the dialog.

Saved HTML files are fully self-contained — they include styles for both light and dark mode and work in any modern browser. The table structure is preserved, so you can navigate them with a screen reader the same way you do inside the app.

Tip: If you want to share an analysis with a sighted colleague, save it as an HTML file. It renders cleanly in any browser with both light and dark mode support, and the screen-reader-navigable headings and tables are already in place.
